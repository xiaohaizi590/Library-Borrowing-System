## 解决的问题
项目中异常散落在各 Service 和 Controller，直接抛给前端，响应格式不统一，且可能泄露内部信息。

## 核心思路
- 用 `@RestControllerAdvice` 拦截所有 Controller 抛出的异常
- 统一包装成项目已有的 `Result<T>` 格式返回
- 分层处理：业务异常 → 参数校验 → 权限异常 → 兜底

## 处理链路

```
Controller 执行
    │
    ▼ 抛出异常
GlobalExceptionHandler 拦截
    │
    ├─ BusinessException          → Result.error(resultCode)
    ├─ MethodArgumentNotValidException → Result.error(400)
    ├─ AccessDeniedException      → Result.error(403)
    └─ Exception (兜底)            → Result.error(500)，不暴露堆栈
```

## 2. DTO/Entity 映射层（Mapper 层）

### 解决的问题
Service 层中内联 DTO 转换逻辑，`convertToDTO()` 方法重复出现，代码冗余且维护成本高。

### 核心思路
- 为每个 Entity 创建独立的 Mapper 类，用静态 `toDTO()` 方法统一转换
- Service 层通过 `XxxMapper.toDTO()` 调用，不再内联转换逻辑

### 创建的文件

| Mapper | Entity → DTO | 作用 |
|--------|-------------|------|
| [UserMapper.java](../server/src/main/java/net/togogo/mapper/UserMapper.java) | `User` → `UserDTO` | 映射 id、username、email、phone、createTime、role |
| [BookMapper.java](../server/src/main/java/net/togogo/mapper/BookMapper.java) | `Book` → `BookDTO` | 映射全部图书字段 |
| [BorrowRecordMapper.java](../server/src/main/java/net/togogo/mapper/BorrowRecordMapper.java) | `BorrowRecord` → `BorrowRecordDTO` | 额外计算 `overdueDays`（逾期天数） |

### 关键点
1. 使用 **静态方法**，无需注入，调用简单
2. 空安全处理：`if (entity == null) return null`
3. BorrowRecordMapper 在转换时附带业务逻辑（计算逾期天数），属于 **含逻辑的映射**
4. Service 层从内联转换改为统一调用后，代码量减少且职责清晰

## 3. 借阅逻辑独立拆分（BookRecordService）

### 解决的问题
`BookService` 中混合了图书管理（CRUD）和借阅管理（借书/还书/续借）两种职责，违反了单一职责原则。

### 核心思路
- 拆出 `BookRecordService` 接口 + `BookRecordServiceImpl` 实现
- 拆出 `BookRecordController` 对应 Controller 层
- 原 `BookService` 只保留图书本身的 CRUD 方法

### 拆出的方法
- `borrowBook()` — 借书（扣减库存、创建借阅记录）
- `returnBook()` — 还书（恢复库存、更新记录状态）
- `renewBook()` — 续借（延长到期日、限制最多续借 2 次）
- `getBorrowRecordsByUser()` — 按用户查询借阅记录
- `getBorrowRecordsByBook()` — 按图书查询借阅记录
- `getOverdueRecords()` — 查逾期记录
- `getAllBorrowRecords()` — 查全部记录

### 关键点
1. 借阅操作是 **事务性** 的：借书需要同时减库存 + 创建记录，用 `@Transactional` 保证原子性
2. 缓存逐出使用 `@CacheEvict(value = {"borrowRecords", "books", "books:all"}, allEntries = true)`，涉及多个缓存区域
3. 续借限制 `renewCount >= 2` 时抛出 `MAX_RENEW_COUNT_EXCEEDED` 异常

## 4. Java 21 升级 + 虚拟线程

### 解决的问题
项目原本使用 Java 17，无法享受到 Java 21 的虚拟线程特性。虚拟线程能大幅降低 IO 密集型场景（数据库查询、Redis 操作）的线程开销。

### 核心思路
- 升级 JDK 版本从 17 → 21（修改 `pom.xml` 的 `<java.version>`）
- Spring Boot 3.2 原生支持虚拟线程，只需一行配置即可开启

### 操作步骤
1. 在父 `pom.xml` 中将 `<java.version>` 从 `17` 改为 `21`
2. 在 `application.yml` 中添加：
   ```yaml
   spring:
     threads:
       virtual:
         enabled: true
   ```

### 原理
- Spring Boot 3.2+ 检测到 `spring.threads.virtual.enabled=true` 后，自动将 `TomcatProtocolHandler` 替换为虚拟线程版本
- 每个 HTTP 请求跑在虚拟线程上（而非操作系统线程），数量可达百万级
- 对于图书管理系统这种 **IO 密集型** 场景（大量数据库查询、缓存操作）提升明显

### 注意事项
- 虚拟线程不适合 **CPU 密集型** 计算（大量计算任务反而会降低性能）
- 不要在虚拟线程中调用 `synchronized` 或 `Thread.sleep()` 阻塞平台线程
- 现有代码无需改动，Spring Boot 自动接管

## 关键点
1. 用 `@RestControllerAdvice` 而非 `@RestController`
2. 返回 `Result<T>` 而非 `ResponseEntity`，和项目现有格式一致
3. BusinessException 直接复用 `Result.error(resultCode)`，自动映射状态码和消息
4. 兜底异常不暴露 `ex.getMessage()`，只返回固定的"系统错误"

## 踩坑记录

### 1. 异常处理器顺序陷阱
- `Exception.class` 兜底必须写在最后，否则会覆盖前面的精细化异常（如 `BusinessException`）
- Spring 匹配规则：**精确匹配优先于通用匹配**，不会按方法书写顺序执行，但如果两个异常有继承关系（比如 `NestedException extends BusinessException`且没有专门的处理方法），会被父类处理器吃掉

### 2. 参数校验异常缺少具体字段信息
当前实现只返回 400 状态码：
```java
@ExceptionHandler(MethodArgumentNotValidException.class)
public Result<Void> handleMethodArgumentNotValidException(...) {
    return Result.error(ResultCode.BAD_REQUEST);
}
```
- 前端只知道"请求参数有误"，但不知道哪个字段错了
- 改进方案：从 `e.getBindingResult().getFieldErrors()` 提取字段名和错误消息返回

### 3. 兜底异常没打日志
```java
@ExceptionHandler(Exception.class)
public Result<Void> handleException(Exception e) {
    return Result.error(ResultCode.INTERNAL_ERROR); // ⚠️ 没打日志
}
```
- 生产环境一旦触发兜底，只能看到前端返回了 500，后台没有任何堆栈信息
- 开发期间建议加 `log.error("unexpected error", e)`，否则线上问题无法排查

### 4. @PreAuthorize 权限异常的正确捕获
- `AccessDeniedException` 由 Spring Security 的 `ExceptionTranslationFilter` 抛出
- 如果 Security Filter Chain 在 `@ControllerAdvice` 生效之前就处理了异常，则无法被捕获
- 当前实现中能正常捕获，说明 Filter 顺序没问题，但要注意如果调整了 Security 配置可能需要重新验证
