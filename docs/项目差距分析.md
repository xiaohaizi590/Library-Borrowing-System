# nacon7 图书管理系统 — 企业级原则差距分析

> 对照现代 Java 企业级核心原则，逐条审查当前项目，定位缺失项。
> 五大原则：分层架构、微服务架构、高并发与高性能、高可用与弹性、安全第一。

---

## 1. 分层架构

| 原则要求 | 项目现状 | 差距 |
|---------|---------|------|
| 经典三层架构 | ✅ Controller → Service → Repository 已完整 | — |
| **全局异常处理（@ControllerAdvice）** | ❌ 不存在 | 异常散落在各 service，直接抛给前端，响应格式不统一，且可能泄露内部信息 |
| **DTO/Entity 映射层** | ❌ Service 层直接做转换 | 缺少独立 Mapper 层，`convertToDTO()` 与 `convertToUserDTO()` 代码重复 |
| **业务逻辑边界清晰** | ❌ 借阅逻辑写在 `BookService` 中 | 应拆出 `BorrowRecordService` 独立职责 |
| **领域驱动设计（DDD）** | ❌ Entity 内含 `@PrePersist` / `@PreUpdate` 校验逻辑 | 领域逻辑散布在实体回调中，核心领域不纯净 |

---

## 2. 微服务架构

| 原则要求 | 项目现状 | 差距 |
|---------|---------|------|
| 业务能力拆分 | ⚠️ 单应用单体 | 单体可接受，但模块边界不清晰 |
| **API 网关** | ❌ 不存在 | 无统一路由、鉴权、限流入口 |

> 当前为单体应用，微服务拆分不是必须的，但模块内聚性有提升空间。

---

## 3. 高并发与高性能

| 原则要求 | 项目现状 | 差距 |
|---------|---------|------|
| **虚拟线程（Virtual Threads）** | ❌ Java 17，未启用 | 未升级 Java 21，无法使用虚拟线程处理高并发 |
| **异步非阻塞编程** | ❌ 全部同步阻塞 | 无 `@Async`、无 `CompletableFuture`、无 `@EnableAsync` |
| **Redis 缓存** | ✅ `@Cacheable` 已使用 | 已使用，但缓存失效粒度粗（`allEntries = true`） |
| **连接池** | ✅ Lettuce pool 已配 | — |
| **批量处理（Batching）** | ❌ 不存在 | 无批量操作（批量导入图书、批量处理逾期记录等） |
| **分页参数保护** | ❌ 无限制 | `page`/`size` 无最大值限制，可被恶意构造大查询攻击 |

---

## 4. 高可用与弹性

| 原则要求 | 项目现状 | 差距 |
|---------|---------|------|
| **熔断降级** | ❌ 不存在 | 无 Sentinel / Resilience4j，故障可能级联扩散 |
| **限流（Rate Limiting）** | ❌ 不存在 | 无 API 级别访问频率限制 |
| **重试机制** | ❌ 不存在 | 失败直接抛异常，无自动重试 |
| **集群部署** | ❌ 不存在 | 无多实例部署配置，单点故障风险 |
| **自定义健康检查** | ❌ 不存在 | 只暴露 Actuator 默认的 `health`、`info` 端点 |
| **登录防暴力破解** | ✅ LoginAttemptService + Redis | 已实现 |
| Docker 容器化 | ❌ 不存在 | 无 Dockerfile |
| Docker Compose 编排 | ❌ 不存在 | 无 docker-compose.yml |

---

## 5. 安全第一

| 原则要求 | 项目现状 | 差距 |
|---------|---------|------|
| JWT 身份认证 | ✅ 已实现 | — |
| **OAuth2.0 协议** | ❌ 不支持 | 只有自签 JWT，无标准 OAuth2 授权流程 |
| **操作审计日志** | ❌ 不存在 | 删除用户、删除图书等关键操作无日志记录，无法事后追溯 |
| **数据加密存储** | ⚠️ 仅密码加密 | BCrypt 加密密码 ✅，但邮箱、手机号等敏感字段明文存储 |
| **HTTPS/TLS** | ❌ 未配置 | 明文 HTTP 传输，中间人可窃听 |
| **Refresh Token 刷新机制** | ❌ 无 | Token 过期只能重新登录，用户体验差 |
| **API 限流** | ❌ 无 | 无接口级别访问频率限制，易被暴力调用 |
| **验证码** | ✅ easy-captcha + Redis | — |
| **参数校验** | ✅ `@Valid` 注解 | — |

## 优先级总览

| 优先级 | 缺失项 | 影响 |
|--------|--------|------|
| **P0** | 操作审计日志 | 数据被删改无法追溯，不合规 |
| **P0** | 全局异常处理器 | 异常信息泄露、响应格式不统一 |
| **P1** | API 限流 | 接口可被无限调用，有安全风险 |
| **P1** | 刷新令牌机制 | Token 过期体验差 |
| **P2** | 虚拟线程 | 并发处理能力未充分利用 |
| **P2** | 异步处理 | 借阅通知等场景需同步等待 |
| **P2** | HTTPS | 传输层无加密 |
| **P3** | 熔断降级 | 单点故障可能级联扩散 |
| **P3** | Docker 容器化 | 部署环境不一致 |
| **P3** | DTO 映射层重复代码 | 维护成本 |
